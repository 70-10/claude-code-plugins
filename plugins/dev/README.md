# dev

`dev` は、Claude Code に開発タスクを任せるときに、作業を小さな工程に分けて進める
Plugin です。各工程が終わるたびに、変更内容やテスト結果が提示されます。内容を確認して
承認するまで次へ進まないため、意図しないまま実装が進むことを防げます。

この Plugin は、次のような場面に向いています。

- 実装内容を工程ごとに確認しながら Claude Code に任せたい
- 大きな変更を、レビューしやすい単位に分けて進めたい
- どのテストが通ったのか、仕様にない判断をしていないか確認したい
- 作業を途中で止めても、同じ場所から安全に再開したい

> [!IMPORTANT]
> `dev` は「何を作るか」を決めるための Plugin ではありません。実装したい内容を文章または
> 仕様ファイルで伝えてから使います。また、push、Pull Request の作成、merge は自動実行しません。

## 5分で始める

### 1. 前提条件を確認する

利用するには、次のものが必要です。

- Plugin を利用できる Claude Code
- `node` コマンド
- 少なくとも1回コミット済みの Git リポジトリ
- 実装したい内容、または要件と完了条件を書いた仕様ファイル

仕様ファイルは必須ではありません。ファイルがない場合は、依頼文に「何を変えるか」と
「どうなれば完了か」を含めてください。

### 2. Plugin をインストールする

Claude Code を起動し、次のコマンドを Claude Code の画面で実行します。

```text
/plugin marketplace add 70-10/claude-code-plugins
/plugin install dev@70-10-plugins
```

### 3. 実装したいリポジトリで開始する

対象リポジトリを開いて Claude Code を起動し、`/dev:flow` に実装したい内容を続けて入力します。

```text
/dev:flow ログインに5回失敗したユーザーを30分間ロックする機能を追加してください。
完了条件は、ロック中のログインを拒否することと、30分後に再試行できることです。
```

仕様ファイルがある場合は、そのパスも伝えます。

```text
/dev:flow docs/specs/account-lock.md の仕様を実装してください
```

### 4. 提案された進め方を確認する

最初に Claude Code が、依頼内容、変更してよい範囲、作業の分け方、実行するテストを確認し、
進め方を提案します。これを**開始前の確認（intake）**と呼びます。この時点では実装ファイルを
変更しません。

提案を確認し、問題なければ「この内容で開始してください」のように明確に承認します。
変更したい場合は、実装を始める前に具体的に伝えてください。

```text
Claude Code: src/auth/** と test/auth/** を変更対象にし、実装を2つに分けます。
             各実装後に関連テストを実行し、最後に全体テストと独立レビューを行います。
             この内容で開始しますか？

あなた:      テストの変更範囲に test/integration/** も追加してください。

Claude Code: 変更範囲を追加しました。この内容で開始しますか？

あなた:      承認します。この内容で開始してください。
```

### 5. 工程ごとの結果を確認する

開始後は、作業の分割、各実装、全体テスト、独立レビューの順に進みます。工程が終わるたびに、
Claude Code は確認に必要な情報をまとめた**確認資料（gate card）**を表示して停止します。

確認資料では、主に次の内容を確認します。

- 何を変更したか
- どのテストを実行し、成功したか
- 仕様に書かれていない点をどのように判断したか
- レビューで問題が見つかったか

問題がなければ `Approve` に当たる明確な承認を返します。直してほしい点があれば
`Request Changes` に当たる具体的な修正内容を返します。

```text
承認する場合: 承認します。次へ進めてください。
修正する場合: Request Changes。空文字の入力テストを追加してください。
```

「たぶん大丈夫」「いいと思う」のような曖昧な返答では先へ進みません。また、「あとは全部進めて」
と依頼しても、以降の確認は省略されません。

### 6. 完了後の操作を選ぶ

すべての工程を承認すると、Claude Code が push、Pull Request の作成、merge などの選択肢を
提示してフローを終了します。`dev` はこれらを自動実行しないため、内容を最終確認してから
次の操作を依頼してください。

## 目的別ガイド

### 仕様ファイルを使って始める

要件、完了条件、対象外の範囲が決まっている場合は、Markdown などの仕様ファイルにまとめて
パスを渡す方法を推奨します。仕様ファイルは、作業の分割、テストによる確認、最終レビューの
判断基準として使われます。

```text
/dev:flow docs/specs/account-lock.md を実装してください
```

### 依頼文だけで始める

仕様ファイルがなくても開始できます。判断基準が曖昧にならないよう、少なくとも次の3点を
含めてください。

```text
/dev:flow
目的: CSVでユーザー一覧をダウンロードできるようにする
完了条件: 管理者だけが実行でき、UTF-8のCSVが返る
対象外: 絞り込み条件の追加と画面デザインの変更
```

不足している情報がある場合、Claude Code は推測して実装せず、開始前に質問します。

### 中断した作業を再開する

同じリポジトリで再び Claude Code を起動し、次のコマンドを実行します。

```text
/dev:flow
```

進行中のフローが見つかると、最後に中断した工程から再開します。新しいフローが重複して
開始されることはありません。

### 結果に修正を依頼する

確認資料が表示されたら、修正箇所と期待する結果を具体的に伝えます。

```text
Request Changes。エラーメッセージにロック解除予定時刻を含め、対応するテストも追加してください。
```

Claude Code は現在の工程を修正して、テスト結果を含む新しい確認資料を提示します。過去の工程まで
戻る必要がある場合は戻り先が提案され、同意後にその工程からやり直します。Git の履歴は巻き戻しません。

### レビューで見つかった問題に対応する

独立レビューで問題が見つかると、問題ごとに根拠と重要度が示されます。それぞれについて、
今回修正するか、理由を記録して受け入れるかを選びます。修正を選ぶと必要な工程へ戻り、修正後に
テストとレビューをやり直します。

## 処理の流れ

通常は次の順番で進みます。小さなタスクでは、作業の分割が省略されることがあります。

1. **開始前の確認（intake）**: 依頼内容、変更範囲、進め方、テスト方法を提案し、同意を待つ
2. **作業の分割（decompose）**: タスクを確認しやすい実装単位に分ける
3. **単位ごとの実装（implement）**: 1単位ずつ実装し、関連テストを実行する
4. **全体の確認（verify）**: 全体テストを実行し、要件と完了条件を満たすか確認する
5. **独立レビュー（review）**: 実装を担当していない別コンテキストで変更内容をレビューする
6. **終了**: 統合方法の選択肢を提示する

開始前の確認と各工程の間には確認ポイント（gate）があり、利用者の明確な承認があるまで
次の工程へ進みません。

## 用語

| 表記 | 意味 |
|------|------|
| 開始前の確認（intake） | 実装前に依頼内容、変更範囲、進め方、テスト方法を確認すること |
| 作業の分割（decompose） | 大きなタスクを、個別に実装・確認できる単位へ分けること |
| 確認ポイント（gate） | 工程の結果を確認し、承認または修正依頼を行うタイミング |
| 確認資料（gate card） | 変更内容、テスト結果、判断、レビュー結果をまとめた資料 |
| 未指定事項の判断（interpretations） | 仕様に明記されていない点について Claude Code が置いた前提や判断 |
| 検証結果（evidence） | テスト結果など、工程が完了したと判断する根拠 |
| レビュー指摘（findings） | 独立レビューで見つかった問題とその根拠 |
| 変更可能な範囲（work roots） | フロー全体でファイルを変更してよい上限。開始前に提案される |

## 安全のための仕組み

- 各工程は、利用者が確認資料を読み、明確に承認するまで進みません。
- 実装中に変更できるファイルは、開始前に同意した範囲へ制限されます。
- 変更内容、テスト結果、未指定事項の判断、承認履歴を記録します。
- フローの状態管理は内部の状態管理機構（bookkeeper）だけが行います。通常の利用では、
  状態管理コマンドを直接操作する必要はありません。
- フローが進行していないとき、Plugin の hook は通常の作業へ干渉しません。

状態は対象リポジトリの `.devflow/` に保存されます。Git で追跡されるのは、次回以降のフローへ
引き継ぐ学習を記録する `.devflow/memory.md` と `.devflow/.gitignore` だけです。

## thinking plugin と組み合わせる

何を作るか、要件や設計を先に整理したい場合は、任意で [thinking plugin](../thinking/) を利用できます。

1. `/thinking:elaborate` で要件や概念設計を整理し、仕様ファイルへ保存する
2. `/dev:flow` に仕様ファイルのパスを渡して実装を開始する

`/thinking:brief` で作成した要求、完了条件、実装方針も仕様ファイルとして利用できます。
`thinking` は `dev` の必須 Plugin ではありません。

## ADR 候補について

仕様にない判断のうち、後から変えにくいもの、背景を知らないと意外に見えるもの、重要な
トレードオフを含むものは、確認資料に `[ADR候補]` と表示されます。`dev` は候補の提示までを行い、
ADR 本文は作成しません。必要に応じてリポジトリのテンプレート、[dr plugin](../dr/)、
`adr-tools` などを使って記録してください。

## Plugin 開発者向けの構成

```text
skills/flow          全体の進行と確認を担当するオーケストレーター
skills/decompose     作業の分割と計画ファイルの作成
skills/implement     単位ごとの実装とテスト
skills/verify        全体テストと要件の確認
skills/review        独立した読み取り専用レビュー
scripts/flow.mjs     状態遷移、承認履歴、書き込み範囲を管理する仕組み
scripts/hooks/       書き込み範囲や確認手順を強制する hook
tests/               node:test によるテストスイート
```
